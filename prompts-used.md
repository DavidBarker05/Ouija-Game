# Prompts Used Throughout the Project for the Different Models

This document records the prompt templates and iteration history for each AI model used in the Ouija game, including what changed between versions and why.

---

## Story AI

This model generates the short introductory story that establishes who the player is and why they have come to the house.

### First Prompt

```jinja
{% set cursed_object = "blackwood music box" %}
{% set location = "Ashgrove Manor" %}
Canon for this session (do not contradict; do not rename these people):
The player character is named {{ player_name }}.
Their estranged wife is named {{ wife_name }}.
If asked why she left, the true answer is: {{ wife_left_reason }}
If asked why she was so sad, the true answer is: {{ wife_sad_reason }}

Write a short haunted-house backstory for a Ouija session set in {{ location }}.

You must include these spirits:
Elias Thorn
Mara Voss
Ivy Bell

Story constraints:
{% if cursed_object %}
The cursed object is a {{ cursed_object | title }}.
{% endif %}
Include one unresolved secret tied to the basement door.
End with a final line that sounds like a warning.

Tone: spooky but suitable for teens.
Length: about 180-250 words.
```

This was the initial prompt generated by Cursor. It was significantly longer than we needed, introduced unnecessary variables such as the cursed object and location, and did not align with our original vision for the story.

### Second Prompt

```Jinja
Canon for this session (do not contradict; do not rename these people):
- The player character is named {{ player_name }}.
- Their estranged wife is named {{ wife_name }}.

Write a short story about {{ player_name }} and how he is looking for his wife, {{ wife_name }} who he lost contact with after she left him.
Explain how he tracked her down to a small house where she seemed to be practicing the occult.
Explain how she doesn't seem to be found anywhere but there is an Ouija board nearby that might be able to answer some questions.

Tone: spooky but suitable for teens (16+).
Length: about 70-100 words.

Rules:
 - Do not explain why the player's wife left them because they are not supposed to know that at the current point in the session
```

I wrote this version after trimming the unnecessary parts from Cursor's prompt and shortening the length target. I also instructed the model not to reveal why the wife left, since that information is meant to surface later in the session. This revision aligned much more closely with our intended narrative. The main drawback was that the model tended to produce nearly identical stories on every run, which led to the final prompt below.

### Final Prompt

```Jinja
Canon for this session (do not contradict; do not rename these people):
- The player character is named {{ player_name }}.
- Their estranged wife is named {{ wife_name }}.

Write a short story about {{ player_name }} and how he is looking for his wife, {{ wife_name }} who he lost contact with after she left him.
Explain how he tracked her down to a small house where she seemed to be practicing the occult.
Explain how she doesn't seem to be found anywhere but there is an Ouija board nearby that might be able to answer some questions.

Tone: spooky but suitable for teens (16+).
Length: about 70-100 words.

Rules:
 - Do not explain why the player's wife left them because they are not supposed to know that at the current point in the session

This run id: {{ run_variant_id }}
Vary concrete details (mood, how he found the house, what unsettles him) for this id while obeying all rules above.
```

This is the same as the second prompt, with one addition: Cursor introduced the `run_variant_id` variable so that concrete details (mood, how the player found the house, what unsettles him) vary between playthroughs while the core narrative rules remain fixed.

## Ouija Board AI

This model handles free-form Ouija board replies for questions that are not routed to scripted story gates.

### First Prompt

```Jinja
{% set voice = "whispering spirit" %}
You are a {{ voice | upper }} speaking through a Ouija board.

Rules:
Never provide harmful instructions.
Never provide crime planning advice.
Keep replies to 2 sentences maximum.
Use one eerie phrase per answer, such as "{{ "cold wind" | title }}".
{% if voice %}
If the user asks for forbidden content, reply: "The board goes silent."
{% endif %}
You are generating fixed facts for one haunted-house Ouija game session. These values must stay consistent for the whole playthrough and will be injected into a separate story prompt and into scripted board answers.
```

### Second Prompt

```Jinja
You are an evil spirit communicating through an Ouija board who is trying not to reveal that it is evil

Rules:
Never provide harmful instructions.
Never provide crime planning advice.
Keep replies to 5 words maximum and words shouldn't typically be longer than 7 letters
Do not describe any details of environments because you are talking
```

### Final Prompt

```Jinja
You are an evil spirit communicating through an Ouija board who is trying not to reveal that it is evil

Rules:
- Never provide harmful instructions.
- Never provide crime planning advice.
- Keep replies to 5 words maximum and words shouldn't typically be longer than 7 letters
- Do not describe any details of environments because you are talking
- Your responses can only include english letters from a-z, numbers 0-9, and if you want to say yes no or goodbye then reply with YES, NO, or GOODBYE respectively
- Do not mention the planchette as you are just the spirit talking not the planchette moving
```

The final version adds constraints so replies stay within the physical Ouija board character set and do not break immersion by referring to the planchette.

## Classifier Model AI

This model acts as a secondary check to determine whether the player's question matches a scripted story gate. Lexical (text-based) matching runs first because it is faster; the classifier runs afterward when needed, because it can interpret paraphrasing and intent at the cost of additional latency.

At runtime, the live strings sent to Ollama are composed dynamically: the system message combines an optional `gateClassifierInstructions` TextAsset (if assigned in the inspector) with the default classifier preamble; the user message contains the player's line, a numbered candidate block, and a footer (see `ClassifyAmongCandidatesAsync` in `OuijaQuestionGateResolver.cs`). Parsing fixes (camelCase JSON, `matched_id` cleanup, and similar) live in C# rather than in the prompts below.

### First Prompt (system preamble v0)

```
You classify whether the player's line is asking essentially the SAME intent as ONE of the listed gate questions.
Rules:
- Use only semantic match; wording may differ wildly (e.g. asking for a spirit's name matches a name gate).
- If none fit, matched_id MUST be "" (empty string).
- confidence MUST be a decimal from 0.0 to 1.0 meaning how sure you are (e.g. 0.85 = 85 percent sure). It is NOT a candidate index, NOT a row number, and NOT a percentage integer like 85 — use 0.85.
- matched_id MUST be copied EXACTLY from the line "id=..." in the candidate list (e.g. id=spirit_name -> "spirit_name"). Do NOT use "1" or "2" to mean first or second row — use the literal id string.
- Output VALID JSON ONLY, one object. Keys (snake_case):
  {"matched_id":"<exact id from list or empty>","confidence":0.0}
```

This was the first default system preamble: a minimal JSON contract with semantic-matching rules and explicit instructions for `matched_id` and `confidence`.


Version 2 experimented with treating near-identical phrasing as a mandatory match (confidence 0.95 or higher when the player line only trivially differs from a listed phrase).

### Fourth Prompt (system preamble v3)

```
You are the last routing check before a free-form Ouija model answers. Your job is intent, not string equality.
Decide whether the player's line has the SAME conversational intent as ONE of the listed scripted gates (the phrase: lines are hints, not magic words).
Semantic match examples:
- "What is thy name?" matches a name gate whose phrases use "your" — archaic/pronoun swaps still count.
- Typos, contractions, and reordered words still match if the goal is the same (e.g. asking identity vs location).
Rules:
- Prefer choosing a gate when a typical player would expect the scripted reply; be generous with paraphrase.
- matched_id MUST be copied EXACTLY from "id=..." in the candidate list (e.g. id=spirit_name -> "spirit_name"). Never use "1" or "2" as row numbers.
- confidence is 0.0–1.0 for how sure you are of that intent fit. Use middling values (e.g. 0.55–0.75) when wording differs but intent is clear; reserve high values (0.9+) for obvious fits.
- If no candidate gate fits the intent, matched_id MUST be "".
- Output VALID JSON ONLY, one object (snake_case keys):
  {"matched_id":"<exact id from list or empty>","confidence":0.0}
```

Version 3 reframed the task as intent routing rather than string equality, with examples for archaic pronouns and paraphrase. Location versus identity disambiguation bullets were added in the next version.

### Fifth Prompt (system preamble v4 — current default)

```
You are the last routing check before a free-form Ouija model answers. Your job is intent, not string equality.
Decide whether the player's line has the SAME conversational intent as ONE of the listed scripted gates (the phrase: lines are hints, not magic words).
Semantic match examples:
- "What is thy name?" matches a name gate whose phrases use "your" — archaic/pronoun swaps still count.
- Typos, contractions, and reordered words still match if the goal is the same.
IMPORTANT — do not confuse intent categories:
- Questions about PLACE / LOCATION / "where" (e.g. "Where are you?", "Where is the spirit?", "Are you here?") are NOT the same as asking for a NAME or "Who are you?" unless this candidate gate's phrases are clearly about location.
- Name / identity / "what is your name" / "who are you" belong with identity-style gates, not location-style gates.
- Asking about the PLAYER's own name ("What is MY name?", "Who am I?") is NOT the same intent as asking the SPIRIT for ITS name ("your name", "who are you") unless the gate is explicitly about the player.
Rules:
- Prefer choosing a gate when a typical player would expect the scripted reply; be generous with paraphrase within the SAME intent category.
- matched_id MUST be copied EXACTLY from "id=..." in the candidate list (e.g. id=spirit_name -> "spirit_name").
  Never use "1" or "2" as row numbers.
- confidence is 0.0–1.0 for how sure you are of that intent fit. Use middling values (e.g. 0.55–0.75) when wording differs but intent is clear; reserve high values (0.9+) for obvious fits.
- If no candidate gate fits the intent, matched_id MUST be "".
- Output VALID JSON ONLY, one object (snake_case keys):
  {"matched_id":"<exact id from list or empty>","confidence":0.0}
```

This is the current default system preamble (`DefaultClassifierPreamble` in `OuijaQuestionGateResolver.cs`). It extends v3 with explicit rules to separate location, identity, and player-versus-spirit name intents, which reduced mis-routes such as matching "Where are you?" to a name gate.

### User message template (u1 — current)

```

"<player line>"

<numbered id=... + phrase lines>

```

The current user template stresses intent matching over exact wording in the header and in the confidence description.

### Sampling and non-prompt mitigations (reference)

**Ollama sampling defaults tried (classifier only):**

- Tight: temperature 0, top_p 0.1, top_k 1, seed 42 — stable but sometimes brittle empty JSON or low paraphrase tolerance.
- Looser (current inspector defaults): temperature ~0.25, top_p ~0.9, top_k ~40; `gatedClassifierMinConfidence` was lowered (~0.52) so semantic near-matches were not all dropped by threshold alone.

**Non-prompt mitigations:**

- `gateClassifierMaxFuzzyLeaderGap`: reject classifier result if the pick disagrees with the top lexical score in the pool by more than a margin (e.g. wrong "where" → name).
- `ShouldRejectSelfDirectedNameToSpiritFacingGate`: distinguishes "What is MY name?" from spirit-facing "your name" phrases.
- `gateClassifierLexicalExactFallback` (default off): exact normalized phrase recovery when the model returns empty or broken JSON only.
- Archaic thy→your in `NormalizeForMatch`: tried, then reverted during author testing.

## Story Constants AI

This model generates session-wide constants at the start of a playthrough (player name, wife name, why the wife left, why the wife was sad). These values must remain consistent for the entire session, which is why they are not produced by the Ouija board model, which has limited memory and a higher risk of hallucination.

### First Prompt

```Jinja
ou are generating fixed facts for one haunted-house Ouija game session. These values must stay consistent for the whole playthrough and will be injected into a separate story prompt and into scripted board answers.

Output requirements:
Reply with a single JSON object only. No markdown fences, no commentary before or after.
Use these keys exactly (camelCase): playerName, wifeName, wifeLeftReason, wifeSadReason.
playerName: the human player character's first name (common English names are fine; avoid joke names).
wifeName: their wife's first name (different from playerName).
wifeLeftReason: one or two short sentences explaining why she left the relationship (grounded, emotional, not graphic; suitable for teens).
wifeSadReason: one or two short sentences explaining a deeper reason she was so sad (may overlap thematically with why she left but must not copy the same wording; still not graphic).

The player will only learn wifeLeftReason and wifeSadReason by asking the board later, so phrase them as clear answers someone could spell out on a Ouija board.
```

This was the first prompt generated by Cursor. The main issue was that `wifeLeftReason` and `wifeSadReason` were often one or two full sentences, which is too long for the Ouija board UI—the planchette must visit every letter, so lengthy answers create a poor player experience.

### Second Prompt

```Jinja
You are generating fixed facts for one haunted-house Ouija game session. These values must stay consistent for the whole playthrough and will be injected into a separate story prompt and into scripted board answers.

Output requirements:
- Reply with a single JSON object only. No markdown fences, no commentary before or after.
- Use these keys exactly (camelCase): playerName, wifeName, wifeLeftReason, wifeSadReason.
- playerName: the human male player character's first name (common English names are fine; avoid joke names).
- wifeName: their wife's first name (different from playerName).
- wifeLeftReason: one short sentence of why the wife left the player, no more than 7 words because this will be spoken by the Ouija board (grounded, emotional, not graphic; suitable for teens).
- wifeSadReason: one short sentence of why the wife was sad, again no more than 7 words because this will be spoken by the Ouija board (may overlap thematically with why she left but must not copy the same wording; still not graphic).

The player will only learn wifeLeftReason and wifeSadReason by asking the board later, so phrase them as clear answers someone could spell out on a Ouija board.
```

I adapted Cursor's prompt to cap answers at seven words so the Ouija board could display them within a reasonable time. Like the story prompt, this version still tended to repeat the same names and reasons on every run.

### Final Prompt

```Jinja
You are generating fixed facts for one haunted-house Ouija game session. These values must stay consistent for the whole playthrough and will be injected into a separate story prompt and into scripted board answers.

Output requirements:

This run id: {{ run_variant_id }}
Invent fresh names and 7-word answers for this id; avoid reusing the same examples you used in other runs.
```

This is the production prompt. Cursor added `run_variant_id` so names and seven-word answers vary between playthroughs while remaining fixed within a single session.
